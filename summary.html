Ford LENS — Pre-Meeting Summary

Benefits, Issues
& Aurora Explained

A concise reference covering what this system does well, what its limitations are, the concerns raised during review, and how we've addressed each one — plus a plain-English explanation of the Aurora Serverless v2 database migration.

± Benefits & Limitations

Benefits
  • Spiders self-heal without anyone being paged Failures are detected, diagnosed, and fixed automatically — before anyone on the team notices
  • Classifier-first approach is clean and predictable Claude identifies the failure type first (anti-bot / structure change / code error) then applies the right fix strategy — no guessing
  • Every fix is a code commit with a PR Full audit trail — engineers can review what Claude changed before it merges to main
  • No new always-on servers needed The healer only spins up when a spider actually fails — zero idle cost between Monday runs
  • Saves ~£155/month vs current infrastructure Mostly driven by the Aurora Serverless v2 migration — database cost drops from ~£165 to ~£30/mo
  • Scales naturally as more spiders are added Adding a new spider requires no changes to the healing system — it picks it up automatically
Limitations
  • Claude cannot create brand new spiders If a new forum domain needs to be added, that still requires a human to build the spider from scratch
  • Login walls stop the healer completely If a forum adds authentication, Claude has no way to obtain or use credentials — manual intervention required
  • Full forum engine changes need human review If a site moves from phpBB to a completely different platform, the spider logic needs rethinking beyond selector fixes
  • Sites fully offline cannot be healed If a domain has DNS failure or permanent 503, there is nothing to fix — Slack alert fires and it's flagged for review
  • Claude costs money per heal attempt ~£0.35 per attempt using Claude Sonnet — weeks with no failures cost £0, but a bad week with many failures adds up

! Issues Raised & How We've Addressed Them

Timing
Can the agent fix a spider within 15 minutes?
Raised by Danny — concern about Lambda timing window
With Lambda, the full cycle (cold start → Claude reasoning → fetch page → generate fix → commit → GitHub Actions build → ECR push → ECS rerun) sits at 8–12 minutes in normal cases but can brush 15 minutes if the target forum page is slow or Claude needs multiple bypass attempts. Tight but feasible — however it leaves no margin.
Solution

We're going with ECS over Lambda for the healer. Running the MCP Healer as an on-demand Fargate task (not always-on) eliminates the cold start and gives us a reliable ~8 minute cycle. A hard 15-minute timeout is implemented at two levels: the Claude loop has a wall-clock cutoff that stops gracefully and fires a Slack alert; the ECS task definition has a stopTimeout as a hard backstop so nothing ever hangs regardless.

Repo Safety
The agent making changes to the repo — how do we limit what it can touch?
Raised by Danny — concern about agent scope in fluency_scraping repo
If Claude is given a GitHub token with full repo write access, it could theoretically modify pipelines.py, settings.py, workflow files, or any other part of the codebase — not just spiders. A bug in Claude's reasoning or a badly generated fix could break the Monday pipeline for all 44 spiders at once.
Solution — Three-layer isolation

1. Dedicated branch: Claude commits to a branch called ford-lens-auto-heal only — never to main directly.

2. Path restriction: GitHub branch protection rules restrict the token to car_forums/spiders/** only. Claude physically cannot write to pipelines.py, settings.py, middlewares.py, or any workflow file.

3. PR before main: Every fix becomes a pull request. The ECS rerun validates it on the branch image first. Only after a human approves does it merge to main and go into the Monday production run.

Cost Control
What if the healer keeps retrying and running up the Claude API bill?
General concern — uncapped AI API spend
Without a retry cap, a spider that Claude consistently cannot fix (e.g. a site that permanently added a login wall) would keep triggering heal attempts every time it runs on Monday, each one costing ~£0.35 and burning 15 minutes of ECS time.
Solution

Hard cap of 3 heal attempts per spider per week. After 3 failed attempts, the spider is marked as needs_human_review, a Slack alert fires, and no further Claude API calls are made until an engineer resets it. Maximum weekly Claude spend is bounded regardless of how many spiders break.

Dashboard
Should the dashboard be on GitHub Pages (free) or ECS (always-on)?
Open question — response time vs cost tradeoff
GitHub Pages serves static files for free but the API calls that power the Run and Heal buttons go through Lambda, which has a ~1–2 second cold start on first use. ECS keeps the API always-on and instant but adds ~£28/month in idle server cost.
Pending decision

Given the dashboard is used roughly once a week (Monday after the run), the 1–2s cold start on Lambda is acceptable and the £28/month saving is real. Our lean is GitHub Pages + Lambda. If Danny wants instant response for the buttons, we switch to always-on ECS — the rest of the architecture is identical either way.

AI How Claude Actually Fixes a Broken Spider

Claude doesn't just retry the spider — it reads the live website, understands what changed, writes corrected code, and verifies the fix works before committing anything. Here's the exact sequence for each failure type.

Anti-Bot Block Cloudflare, Akamai, 403, CAPTCHA, rate limiting
1
Detect block type
Reads HTTP status + HTML. Identifies if it's Cloudflare, Akamai, DataDome, captcha, or plain 403
2
Try strategies
Tests curl_cffi → scrapling → playwright → proxy rotation in sequence. Stops at first success
3
Read spider code
Fetches current spider from GitHub. Identifies which middleware is in use and what needs to change
4
Rewrite spider
Switches the downloader middleware class, updates settings flags, adjusts request headers if needed
5
Run test suite
Runs summary crawler checks. Confirms >0 posts scraped, fields populated. Only commits if tests pass
Structure Change Site redesign, new CSS classes, moved HTML elements, pagination changes
1
Fetch live HTML
Downloads the current page using the working bypass strategy (50k char cap for context window)
2
Analyse structure
Reads the HTML DOM. Finds where post_author, post_content, post_date, thread_title, and next_page now live
3
Read old selectors
Fetches spider from GitHub. Compares old XPath selectors against new HTML structure to understand what broke
4
Write new XPaths
Generates new XPath selectors for all 5 fields. Mentally tests each one against the fetched HTML before writing
5
Rewrite spider
Replaces old selector strings with new ones. Commits the complete updated spider file
6
Run test suite
Summary crawler confirms fields are populated and not empty. Only commits if all checks pass
Code Exception AttributeError, NoneType, index out of range, unhandled exception in spider logic
1
Read traceback
Reads the full error log and traceback. Identifies exactly which line and method caused the exception
2
Fetch spider
Gets full source from GitHub. Reads the method that failed alongside the traceback to understand the root cause
3
Confirm site works
Tests URL accessibility first — rules out anti-bot as the underlying cause before touching code
4
Patch the bug
Fixes the specific method — adds null guards, fixes selector fallbacks, handles edge cases the original code missed
5
Run test suite
Summary crawler confirms no exceptions raised and output data is valid before the fix is committed

✓ Summary Crawler Test Suite — How We Know the Fix Worked

Each spider has a lightweight summary crawler — a short test script that scrapes just 1–2 pages and validates the output against a checklist. Claude runs this before committing any fix. If the summary crawler fails, Claude keeps trying. It only commits to the repo when all checks pass.

What each summary crawler checks
✓
Items scraped > 0
At least one post returned from the test page. Zero items = fix didn't work.
✓
post_author is not empty
Author field extracted and is a non-empty string. Not None, not whitespace.
✓
post_content length > 10 chars
Content field has real text, not just whitespace or an empty tag.
✓
post_date is a valid date
Date field parses as a real datetime. Not a raw string like "2 hours ago" that wasn't handled.
✓
thread_title is not empty
Thread title extracted correctly from the page.
✓
next_page link found
Pagination working — spider can navigate beyond page 1 without getting stuck.
✓
No exceptions raised
The test run completes cleanly — no AttributeError, NoneType, or unhandled exceptions in the log.
How it fits into the heal loop
1.Claude generates a fix
2.Runs summary crawler (1–2 pages only, ~30s)
3.All 7 checks pass → commit to branch
✗Any check fails → Claude tries a different approach
✗Still failing after 3 attempts → Slack alert, human review
Why this matters

Without the test suite, Claude could commit a fix that appears valid (no syntax errors, no exceptions during import) but produces empty or malformed data silently. The summary crawler is the gate that proves the fix actually extracts real content from the live site before anything is committed or deployed.

Summary crawlers are written once per spider

Each of the 44 spiders gets a companion test script during Phase 1. It's ~30 lines of Python — just enough to scrape one thread page and assert the 7 fields. These live in car_forums/tests/ and run in under a minute. Claude calls them as a tool during healing. They can also be run manually to sanity-check any spider at any time.

DB Aurora Serverless v2 — What It Is & Why We're Migrating

The Simple Explanation
The current ford-scraping database is like leaving a car engine running 24 hours a day so it's ready when you need it on Monday. Aurora Serverless v2 is the same car but it automatically turns off when parked and starts itself up before you need it — saving fuel the other six days without any change to how you drive it.
Mon
Pipeline runs
44 spiders write data. Aurora scales up to handle the load in seconds.
Tue
Scales down
No connections. Aurora scales to minimum — costs ~£0.04/hr idle.
Wed–Sat
Near zero cost
Nothing running. ~6 days × £0.04/hr = ~£5.76 for the idle period.
Next Mon
Ready instantly
Scales back up in seconds when the pipeline starts. No downtime.
Current — RDS db.m5.2xlarge (always-on)
  • Runs 24 hours a day, 7 days a week
  • 8 vCPU, 32 GB RAM provisioned at all times
  • CPU sitting at under 2% on Tue–Sun
  • 0 database connections for 6 out of 7 days
  • Paying for peak capacity even at idle
  • Cost: ~£165/month regardless of usage
Proposed — Aurora Serverless v2
  • Scales down to 0.5 ACU when no connections exist
  • Scales up automatically in seconds on Monday
  • Same PostgreSQL — zero code changes to any spider
  • Same connection string format — swap the endpoint only
  • No performance loss on Monday when it counts
  • Cost: ~£30/month (mostly Monday activity)
Migration Process — ~2 Hours, Zero Code Changes
1
Snapshot
Take a snapshot of the current ford-scraping RDS. Full backup, no data loss.
2
Restore
Restore the snapshot as a new Aurora Serverless v2 cluster. All data intact.
3
Swap endpoint
Update the DB_HOST environment variable in ECS. No SQL changes, no code changes.
4
Verify & retire
Run one Monday cycle on Aurora. Confirm all data writing correctly. Decommission old RDS.
