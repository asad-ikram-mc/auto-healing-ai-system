Option A — Always-On ECS

Django REST API and MCP Healer run as permanent ECS Fargate services behind a load balancer. Always responsive with no cold starts. Suited for teams that need real-time heal progress or higher dashboard traffic.

What We're Building

44 Scrapy spiders run every Monday. When a spider fails — due to anti-bot blocks, site structure changes, or code exceptions — it currently requires manual investigation. This proposal adds a live monitoring dashboard and an AI auto-healer that diagnoses and fixes failures automatically.

Auto-Heal Flow

When a spider closes with zero items or an unhandled exception, the healing pipeline fires automatically. The cycle from failure detection to a fixed, re-running spider requires no human involvement in either deployment option.

End-to-End — Spider Failure to Resolution

What Claude fixes automatically

  • Structure changes — fetches live HTML, generates new XPath selectors for author, content, date, pagination
  • Anti-bot blocks — tries curl_cffi, scrapling, playwright, proxy rotation in sequence until one works
  • Code exceptions — reads spider source from GitHub, identifies root cause, patches the broken section

What stays exactly the same

  • All 44 spiders run unchanged every Monday via EventBridge
  • Same ECR image, same ECS Fargate task definitions, same S3 output format
  • Slack alerts continue to fire for failures Claude cannot resolve automatically

Claude MCP Tool Suite

Claude runs in an agentic loop — it reads context, decides which tool to call next, calls it, reads the result, and continues until the spider is fixed or it cannot resolve the issue. Below are all 9 tools available to Claude, grouped by category. Claude decides which to call based on the failure type it classified.

All 9 MCP Tools — Endpoints, Parameters & Purpose
Category Tool Parameters What It Does
Diagnostic classify_failure() error_log, items_scraped, status_code Reads the spider error log and item count. Returns failure_type: anti_bot / structure_change / exception / zero_items with a confidence score. Always the first tool called.
test_url_strategy() url, strategy Tests whether a URL is accessible using a specific strategy: curl_cffi, scrapling, playwright, or requests. Returns accessible: true/false + block_type (cloudflare / akamai / captcha / 403 etc).
try_all_strategies() url Runs all four strategies in sequence (curl_cffi → scrapling → playwright → requests) and returns which ones succeed. Used when the best bypass approach is unknown. Stops at first success.
fetch_page_html() url, strategy Fetches the live HTML of a forum page (capped at 50k chars for context window). Claude then inspects the structure to identify new XPath selectors for post_author, post_content, post_date, thread_title, and next_page.
update_heal_status() heal_id, status, message, strategy_used, commit_sha PATCHes the HealAttempt record in Aurora. Called throughout the loop so the dashboard shows live progress. Final call sets status to success, failed, or partial.
GitHub get_spider_code() spider_name Fetches the full Python source of the spider from GitHub via the Contents API. Claude reads this to understand the current selector logic, middleware used, and where the failure likely occurred before writing a fix.
commit_spider_fix() spider_name, new_code, commit_message Commits the complete fixed spider file to the main branch via the GitHub API. This automatically triggers deploy.yml which rebuilds the ECR image. Claude always commits the full file, not a partial diff.
ECS trigger_ecs_run() spider_name Starts a new ECS Fargate task for the named spider using the scrapy-cluster and scrapy-task definition. Passes spider_name as a CMD override so the same shared ECR image runs any spider. Returns the task ARN.
poll_task_status() task_arn, timeout_seconds Polls the ECS DescribeTasks API every 30s until the task reaches STOPPED state. Returns SUCCESS, FAILED, or TIMEOUT with the container exit code and stop reason.

Loop configuration

  • Maximum 20 tool calls per heal session — prevents runaway API spend
  • Model: Claude Sonnet (claude-sonnet-4-5) — handles all tool tasks well
  • Each heal session runs in a background task — dashboard polls for progress
  • If Claude reaches the limit without fixing: status → failed, Slack alert fires
  • Max 3 heal attempts per spider per week — hard cap on Claude API cost

What Claude never does automatically

  • Create a brand new spider for a domain that wasn't previously covered
  • Handle a site that has added a login wall or requires credentials
  • Fix a spider where the forum engine has completely changed (e.g. phpBB → Discourse)
  • Retry if the target domain is fully offline (HTTP 503 / DNS failure)
  • In all these cases: status → failed, Slack alert, manual review needed
